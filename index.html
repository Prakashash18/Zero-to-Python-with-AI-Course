Python with AI

What is AI — and why it makes coding easier

An AI helper (like Claude, ChatGPT, Gemini or Copilot) does two things for a beginner: it turns plain English into Python code, and it explains code back to you in plain English. You don't have to memorise syntax. You learn to ask well and read the answer.

Think of it as a fast junior partner, not a magic oracle — it sometimes gets things wrong, which is exactly why we still learn to read code. That's what the rest of Step 1 is for.

How to ask: writing a good code prompt

A good prompt is the whole skill. Three habits get you most of the way:

  1. Be specific and complete. Say the goal, the inputs, and the output you want. "Count steps and show the total" beats "make a counter".
  2. Give context and purpose. Tell it you're a beginner and why you need it, so it explains at the right level.
  3. Ask one thing at a time. Get it working first, then ask for a single improvement.
Weak prompt
"make a counter"
Strong prompt
"Write a Python program that counts steps and prints the total. I'm a beginner — explain each line simply."
Two prompts worth memorising. When you're stuck: "Explain this code line by line, like I'm new."  and  "What does this error mean and how do I fix it?" These remove the biggest beginner fear — getting stuck alone.

Code Fast

micro:bit step counter

Put your code on a real device

The micro:bit is a tiny computer with a built-in accelerometer — a motion sensor, the same kind in your phone. We'll use it to count steps. Same Python ideas from Step 1, now running in your hand.

1 · Connect it

Plug the micro:bit into your laptop with the USB cable. Open the MakeCode editor, click Connect and follow the pairing prompt. Flash the built-in flashing-heart example first — once that shows on the LEDs, you know everything works.

Open MakeCode ↗Python editor ↗
Beginner tip. Build in MakeCode blocks first, then switch to the Python view. Seeing the same program as blocks and as Python is the moment it clicks: blocks are Python.

2 · Build the step counter

Here it is — and every line is a Step 1 idea in the wild. Flash it, strap the micro:bit to your ankle or hold it, and walk around the room.

from microbit import *

steps = 0              # the variable (a box)
while True:            # the loop (runs forever)
    if accelerometer.was_gesture('shake'):  # the if (a step happened)
        steps += 1     # add one
        display.show(steps)

Does the count match your real steps? Almost never exactly — that's a great thing to talk about, and it's what the next part fixes.

3 · Make it better

Pick any of these. Ask your AI helper to write one, then read its explanation:

  • Reset button: if button_b.is_pressed(): steps = 0
  • Show distance (reuse the Step 1 machine): display.scroll(steps * 0.75)
  • More accurate — swap the shake for a raw threshold: if accelerometer.get_y() > 1500: and tune the number.

Build with Data

Step data analyser

From a device to a pile of numbers

In Step 2 your micro:bit produced step numbers. Real trackers like Fitbit produce thousands of them. Now you learn to make sense of them — using the same lists, loops and functions from Step 1, no fancy libraries needed to start.

1 · Get the files

Download a clean week (and a full month) of sample data plus a starter script. Open them in Trinket to code along, or open the notebook in Colab for real charts.

↓ step_data.csv ↓ starter.py Open Trinket ↗ Open in Colab ↗

2 · Analyse it

Start with the data as a list (the idea we previewed in Step 1), then squeeze insights out of it. The big numbers first:

days  = ["Mon","Tue","Wed","Thu","Fri","Sat","Sun"]
steps = [6200, 8100, 6400, 13420, 7300, 12500, 11200]

print("Total:", sum(steps))
print("Average:", round(sum(steps)/len(steps)))
print("Best day:", max(steps))

Then a goal check (the if) and a text bar chart (the for loop) — no chart library required:

for i in range(len(days)):
    bar = "#" * (steps[i] // 1000)
    print(days[i], bar, steps[i])

3 · End with the insight, not the code

"Your most active day was Sunday; weekends beat weekdays. What would you change next week?" That's the difference between running code and building with data — the numbers become a decision.
